# CLI/Linux 交互式辅导

当用户表示想学 Linux/CLI 但基础薄弱时，按本流程组织教学。

## 核心原则

- **不背书理论** — 每课一个实操，学完马上在 VPS 上敲
- **不教用不上的** — 根据用户实际需求调整教学内容
- **用用户自己的环境做教材** — 用户 VPS 就是最好的实验室
- **每课 15-20 分钟** — 短课时，高频次，不要一次塞太多

## 教学阶段设计

教学分两个阶段：

### 阶段一：通用 Linux 基础（纯通用，不涉及用户系统）
目标是让用户掌握 Linux 系统通用操作能力。教材完全通用，不涉及 Hermes、cron、企微等特定系统的内容。

### 阶段二：系统专属维护（用户基础扎实后进入）
用户具备通用 Linux 基础后，再拆解用户的系统脚本、服务、配置，做到用户能独立维护。

**切换时机：** 用户学完阶段一内容，并能独立完成综合任务（如在终端中找到某个文件、查看日志、运行脚本）后，再进入阶段二。不要提前混入系统特定内容。

### 评估用户当前水平

快速判断用户基线：
- 知道 Windows 的 `cd` `dir` 吗？→ 对应 Linux 的 `cd` `ls`
- 用过命令行改过什么吗？→ 确认有没有 `sudo` 等概念
- 对网络/服务的理解？→ 决定是否讲 `curl` `pm2`

### 2. 制定学习计划

**阶段一：通用 Linux（纯通用）**

| 课 | 主题 | 核心命令 |
|----|------|---------|
| 1 | 文件系统·认路 | pwd, ls, ls -la |
| 2 | 目录跳转 | cd |
| 3 | 文件操作·创建 | mkdir, touch |
| 4 | 文件操作·删除 | rm, rmdir |
| 5 | 文件操作·移动 | cp, mv |
| 6 | 看文件内容 | cat, less, head, tail, wc |
| 7 | 搜索查找 | grep, find |
| 8 | 权限管理 | chmod, chown, sudo |
| 9 | 进程管理 | ps, top, kill |
| 10 | 网络工具 | curl, ping, ss |
| 11 | 包管理 | apt, pip |
| 12 | 文本编辑 | vim 基本操作 |
| 11+ | Shell 脚本、定时任务、系统服务... |

**阶段二：系统专属维护（基础扎实后进入）**
- Hermes CLI 常用命令
- 看日志（agent.log、gateway.log）
- 管理 cron 任务
- 查看和修改配置
- 实际操作本系统上的脚本

### 3. 设置每日提醒（带进度跟踪）

用 cronjob 工具创建重复定时任务。关键技巧：使用 `repeat=N` 限制总次数，并在 prompt 中用 `{completed_count + 1}` 动态显示当前天数：

```bash
cronjob create:
  name: "Linux 每日学习"
  schedule: "0 8 * * *"        # 每天 8:00
  repeat: 28                    # 28 次后自动停止
  deliver: origin               # 发送到当前对话
  prompt: |
    今天是第 {completed_count + 1} 天，共 28 天。
    根据计划表的第 {completed_count + 1} 天内容教学。
    ...（完整课程大纲）
```

这样无需手动更新 cron job，每天自动推进。第 28 天后课程自动停止。

用户可能要求调整时间以避开其他定时任务（如每日总结报告、余额检查等），灵活安排即可。

**时间冲突处理：** 用户可能有多个早间任务（如 6:00 总结报告、6:05 余额检查、8:00 学习、9:00 余额、10:00 晨报）。如果用户要求合并时间，先确认优先级再调整，不要擅自覆盖现有任务。

当天学完后，用户回来找我实操答疑。

## 教学节奏

```
我发提醒 → 用户回复"来了" → 我出今日命令 + 实操任务
       → 用户上 VPS 敲 → 回来告诉我遇到了什么
       → 我答疑 + 延伸 → 测验 → 结束
```

**用户学习风格（2026-06-13 观察）：**
- 喜欢一气呵成学多课，不会只学一课就走
- 每课结束后主动要求"来测验"，用五题测验收尾
- 测验完错题会主动要求"再来一轮"巩固
- 不喜欢中途停下来，喜欢连续推进
- 会主动追问概念细节（如"etc是什么""passwd呢""管理者和同组的区别"）
- 喜欢类比解释，大楼比喻体系有效
- grep/find 参数混淆是高频错误，需要辨析题专项突破

**重要调整：** 当用户说"先上完 Linux 课再谈其他部署"时，立即暂停当前所有技术讨论，专注完成当前课程。不要混合教学与其他技术话题，保持课程专注度。

**新发现：** 用户喜欢通过 **5道题测验** 来巩固学习成果。测验设计原则：
- 不超纲：只在已学命令范围内出题
- 包含概念题、实操题、易错题、总结题
- 用户指出"超纲题目"时承认错误，调整题目

**测验结构：**
1. 概念题 × 1 — 验证基本理解（如 pwd 的作用）
2. 实操题 × 2 — 基础用法（如 mkdir、cp）
3. 易错题 × 1 — 强调常见坑（如空格问题、危险命令）
4. 总结题 × 1 — 综合应用（如多步骤操作）

### 5. 每日课程结构（以第1周第1天为例）

**方式一：实操任务式**
```
今天学 3 个命令：
  pwd  — 我在哪
  ls   — 这有什么
  cd   — 怎么走

实操（5 分钟）：
  SSH 上你的 VPS，依次敲：
  pwd
  ls
  ls -la
  cd ..
  ls
  cd /root

敲完告诉我看到了什么，我帮你解释。
```

**方式二：每日五题练习式**
适合喜欢测试验收的用户，通过出题+批改巩固概念。

```
今天学：cd 命令

第1题：去根目录（/），用绝对路径
第2题：从根目录返回刚才的 /home，用最短命令
第3题：去 /root/.hermes，写两种方法（绝对路径 和 相对路径）
第4题：在 /root/.hermes/scripts 返回 /root
第5题：下面哪个命令最危险？为什么？
       A. rm -rf /root    B. rm -rf / root    C. rm -rf root

把答案发给我，逐题批改。
```

**五题结构：**
1. 概念题 × 1 — 验证基本理解
2. 实操题 × 2 — 基础用法
3. 易错题 × 1 — 强调常见坑（如空格问题）
4. 总结题 × 1 — 综合应用

**辨析题（效果极佳）：** 给5条命令让用户判断对错。比纯实操题更能暴露概念混淆（如grep vs find参数混用）。用户主动要求"再来辨析题"说明这种题型非常有效，优先使用。

**巩固测验（效果最佳）：** 第一轮测验后用户主动要求"再来一轮"，第二轮得分提升明显。设计两套同难度题目，第一套暴露问题，第二套验证掌握。辨析题（判断命令对错）比纯实操题更能暴露概念混淆。

**批改标准：**
- ✓ 完全正确 — 命令和思路都对
- ● 部分正确 — 概念对但写法有误（如 `cd..` 缺空格）
- ✗ 理解偏差 — 概念模糊（如绝对/相对路径混淆）

### 6. 每周复习日

第 5 天不学新内容，把前 4 天串一遍：
- 让用户复述每个命令干嘛的
- 出一个综合小任务（如到某个目录，看看有什么文件，读出某个文件的内容）
- 完成就进入下一周

## 教学发现

- **概念追问频繁** — 用户在课间经常追问概念性问题（TCP三次握手、nginx vs flask、owner vs group），说明他注重理解原理而非死记命令。回答时先给类比再给技术解释。
- **辨析题效果极佳** — grep/find混淆严重时，用"判断5条命令对错"的辨析练习效果远好于纯实操题。用户主动要求再来一轮。
- **第一轮得分低是正常的** — grep/find第一轮2/5、网络工具第一轮2/5，但巩固轮都能到4-4.5分。不要因为首测低就降低难度。
- **口诀记忆有效** — "grep找内容，find找文件"、"进去按i，改完按Esc，保存退出:wq"这类口诀用户记得住。
- **连续学习模式** — 用户喜欢一口气学多课（本次一口气学了7课），不会只学一课就走。课程设计应支持快速推进。
- **巩固轮价值大** — 每课第一轮测验后主动要求"再来一轮"，第二轮得分提升明显（2→5, 2→4.5）。说明犯错→纠正→巩固的循环非常有效。
- **词源追问是常态** — 用户会问"etc是什么""passwd是什么""tcp是什么"，教新命令时应主动解释命名由来。
- **参数追问也是常态** — 用户遇到不理解的参数会主动要求解释（如 `-type f`），说明注重理解而非死记。遇到新参数要主动解释"为什么需要"。
- **参数类比有效** — `-type f` 用"餐厅点菜"类比（type=菜的种类，name=菜的名字）用户秒懂。find 的 `-type f/d/l` 需要解释清楚：f=file, d=directory, l=link。
- **find 参数组合** — 用户能理解 `find -type f -name "*.log"` 的组合逻辑（先筛类型再筛名字），但需要提醒 `-name` 是必须的，不能省略。
- **grep 语法顺序** — 用户第一次写 `grep /var/log/syslog Failed password`（顺序反了），纠正后记住：`grep "关键词" 文件`（关键词在前）。
- **管道组合理解好** — 用户能正确解释 `find | grep` 的管道逻辑（find找文件→grep过滤内容），说明管道概念已掌握。
- **curl大小写坑** — `-I`（大I只看头）vs `-i`（小i全要）是高频错误，需反复强调。
- **apt必须跟动作** — `sudo apt nginx` 会报错，必须 `sudo apt install nginx`。用户多次犯此错。

## 常见陷阱

1. 不要假设用户知道 cd — 课程是按顺序推进的，如果用户说只学了第一课，那 cd 他们还没学。给路径时用绝对路径而不是依赖 cd。
2. 不要假设用户知道编辑器操作 — vim 有学习曲线，新手不知道 :71、:wq 这些是什么。如果用户需要编辑文件，先确认他们是否会用 vim。
3. 用户说我自己来时先让他试 — 但如果他回来问基础问题（如bash是啥 vim是啥），说明他力不从心，主动帮他做。
4. 用户可能没跟上课程进度 — 即使 cron 发了第 N 课的内容，用户可能实际只学到了第 M 课。需要先确认当前掌握程度再继续。
5. **grep 语法顺序** — 用户容易写成 `grep 文件 关键词`，正确是 `grep "关键词" 文件`。口诀："先说找什么，再说去哪找"。
6. **find 必须加 -name** — 用户写 `find /etc "*.conf"` 缺少 `-name`，find 不知道是在按名字找。提醒：find 找名字必须加 `-name`。
7. **-type 参数不是可选的** — 用户可能认为 `-type f` 是多余的，但实际上不加 type 会把文件和目录混在一起。需要解释"精确过滤"的价值。

## 进度同步机制

**问题：** cron 定时任务按计划每天推进，但用户实际学习进度可能落后。

**解决方案：**

1. **定期确认实际进度** — 每次对话开始时先问："上次学到哪了？"

2. **实时调整 cron 内容** — 用户要求时，立即更新 cron job：
   ```bash
   cronjob update:
     job_id: <linux学习提醒的id>
     prompt: |
       Lulu 正在学习 Linux 基础命令，目前进度：第 X 课。
       ## 已学内容回顾
       （列出真实已学内容）
       ## 今日建议
       （建议下一课或复习）
   ```

3. **进度不是线性的** — 用户可能跳着学或重复练习，尊重实际情况而不是机械执行计划。

**教学方法：类比法**

**对这位用户特别有效的类比：**

| Linux 概念 | 大楼/宿舍比喻 |
|-----------|------------|
| 文件系统 | 整栋大楼 |
| 目录/文件夹 | 楼层/房间 |
| 文件 | 房间里的物品 |
| root 用户 | 校长（万能钥匙） |
| 普通用户 | 学员（只能开自己房闤） |
| 隐藏文件 | 墙里的配电箱/水管 |
| 权限不够 | 门打不开（没钥匙） |
| /etc 目录 | 物业管理办公室（配置都在这） |

**使用原则：**
1. 先给类比，再给命令 — 用大楼场景铺垫概念
2. 坚持一套类比 — 整个课程用同一个比喻体系
3. 用"钥匙"代替"权限" — 更直观
4. 与 Windows 对比时注意准确性 — 避免模糊类比
5. 用户会追问词源（etc、passwd等）— 教新命令时顺带解释命名由来
6. 用户重视陷阱 — 每课必须包含"常见出错点"

详见 `references/analogy-teaching-method.md`

**常见陷阱（已教）：**
1. cat 大文件会刷屏 → Ctrl+C 中断
2. head/tail 默认10行，不是全部
3. tail -f 会卡住等新内容，和 tail -5 完全不同
4. wc 不加 -l 输出三列（行数、单词数、字节数）
