# 破坏性操作安全规范

执行删除、修改、部署等不可逆操作时的标准确认流程。

## 触发条件

听到以下**关键词**时，必须暂停执行，先追问确认：

### 🔴 破坏性关键词（必须确认）
| 类型 | 关键词 | 确认内容 |
|------|--------|----------|
| 删除类 | 删掉、去掉、清理、清空、删除、卸载、移除、废弃 | 具体路径/范围/彻底删除还是仅停用 |
| 修改类 | 改一下、调整、替换、更新、覆盖、重置 | 改哪里/改成什么样/是否有备份 |
| 执行类 | 开始、做吧、启动、运行、执行、部署、应用 | 具体任务/前置条件/影响范围 |
| 危险类 | 格式化、初始化、重建、刷新、清空 | 双重确认/数据备份/回滚方案 |

## 标准确认话术

> "确认一下：你说**[关键词]**，是指 **[具体选项A]** 还是 **[具体选项B]**？"

### 示例
- 用户："删掉吧" → 追问："是指删掉 V2 半成品（具体路径），还是连生产库一起删掉？"
- 用户："改一下配置" → 追问："是指修改 ~/.hermes/config.yaml 的哪个字段？有备份吗？"
- 用户："开始部署" → 追问："是指部署到生产环境还是测试环境？"

## 安全检查清单

执行前必须确认：
- [ ] **范围明确**：具体文件/目录/资源路径
- [ ] **影响评估**：是否会破坏现有功能/数据
- [ ] **备份存在**：重要操作前是否已备份
- [ ] **回滚方案**：出问题能否恢复

## 血泪教训

**案例：2024-06-07 误删 meme_db/ 生产文件**
- **用户说**：」删掉吧「
- **我理解为**：删掉 V2 半成品
- **实际删除**：V2 半成品 + 生产库 meme_db/
- **原因**：目录结构混乱 + 未追问确认范围
- **损失**：5693张图索引丢失，需重建

**案例：误删 qianfan provider**
- **用户说**：」把 alibaba 的 api 删掉「
- **我理解为**：删掉所有阿里相关配置，包括 qianfan
- **实际删除**：alibaba dashscope + qianfan（qianfan 是百度，不是阿里）
- **原因**：没有区分」公司名「和」provider名「，自作主张扩大删除范围
- **教训**：用户说删 A，只删 A。不要联想扩展到 B。如果不确定范围，先问清楚

**案例：2026-06-09 未经预警重启网关**
- **用户说**：」把之前配的gpt给去掉「 → 我直接删 codex-gpt + 重启网关
- **后果**：微信断开连接，用户侧感知到断联
- **原因**：没想到重启网关 = 断开所有平台连接，没有预警
- **教训**：重启网关前必须说」要重启网关让配置生效，大概断几秒「，而不是直接执行

**案例：2026-06-12 PM2 和 systemd 同时管网关导致无限崩溃**

- **操作**：改完 gateway/run.py 后用 PM2 重启，但没先停 systemd
- **后果**：systemd `Restart=always` 和 PM2 `autorestart=true` 互杀，77次崩溃循环
- **根因**：两个 init 系统抢同一端口，互相拉起→杀进程→再拉起
- **教训**：重启网关前先检查有没有多个管家：
  ```bash
  pm2 list | grep gateway
  systemctl --user status hermes-gateway
  ```
  只能留一个。Lulu 偏好 PM2。

## 🔴 数据通道安全红线（2026-06-12）

**任何情况下，不要用未加密/未认证的方式暴露文件到公网。** 包括但不限于：

- ❌ `python -m http.server --bind 0.0.0.0` — 文件直接对公网可读
- ❌ `nc -l -p 8888 < file` — 裸 TCP 传输
- ❌ 任何监听在 `0.0.0.0` 的无认证 HTTP/FTP 服务

**需要从远程机器读文件时，必须通过安全通道：**
- ✅ Tailscale SSH / 标准 SSH
- ✅ Tailscale Serve（启用 HTTPS + 认证）
- ✅ `scp` 通过 SSH 隧道
- ❌ 绝对不要图快用 HTTP 暴露

**如果 SSH 暂时不可用，优先修 SSH，不要用 HTTP 临时替代。**

## 关于网关重启的补充红线

网关重启（无论 `kill`, `systemctl stop`, 还是正常重启）会导致所有平台连接断开：
- **微信（WeChat）** — 立刻断联，用户侧感知到消息发不出/收不到
- **企业微信（WeCom）** — 重连可能失败（bot_id/secret 问题）
- 新网关启动约需 3-5 秒，平台重建连接需额外 3-10 秒

### 🔴 重启前必须检查：会不会让你失联？（2026-06-12）

**如果当前会话通过微信/企微等网关平台连接，重启会使你中断与我的对话。**

**强制检查流程**：
1. 确认当前会话平台：weixin / wecom / telegram / ...
2. 如果是网关平台：**必须先明确告知用户**「重启会导致微信断联几秒」
3. 等用户说「可以」「搞」后再执行
4. 绝不假设用户知道风险——必须主动说明

**禁止行为（会断联）：**
- ❌ `kill -15 <PID>` — 直接杀 Hermes gateway 进程
- ❌ `kill -9 <PID>` — 暴力强杀
- ❌ `systemctl --user restart` — systemd 方式重启也会断

**正确做法：**
- 先问 Lulu：」要重启网关让配置生效，会断几秒微信，可以吗？「
- 等 Lulu 说「搞/行」→ 再执行
- 重启后检查日志确认平台连接正常

## 例外情况

以下情况可跳过确认（但仍需谨慎）：
- 用户明确指名道姓：`rm -rf /root/.hermes/meme_cache_v2/`
- 用户已在上下文中明确范围："把 V2 的那些图删掉"
- 是用户自己的临时文件（非系统/非共享资源）
