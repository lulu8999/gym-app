# 场景：用户确认后反复重问

## 问题

用户在对话中确认了要做某事（如"加上去"、"搞"、"执行吧"），但经过一次网关中断/上下文压缩后，助手重新询问用户"要不要做X"，导致用户不满。

## 真实案例

```
用户: "加上去，然后识别一下之前我给超级大脑发的图片"
[网关中断]
系统: [System note: ... was interrupted by a gateway interruption]
助手: 服务都恢复了。之前问你的——图片支持要不要现在加上？
用户: "不要再重复问我了，我已经同意了"
```

## 根因

1. 助手在中断恢复后没有读取历史对话中用户已做出的决策
2. 助手把系统 note 当作"新消息"来处理，忽视了之前的对话上下文
3. 助手没有将"用户已同意"这个事实作为不可逆状态来处理

## 解决方案

### 中断恢复时的检查清单

当看到系统中断 note 时，在执行任何操作前：

1. **加载 `hermes-session-recovery` 技能**
2. **扫描本会话（或最近会话）中用户最后几个确认性回应**
   - 寻找关键词："加上去"、"搞"、"可以"、"执行吧"、"发吧"、"ok"、"好"
   - 特别注意用户对"要不要做X"类问题的回应
3. **如果发现用户在中断前已确认某事 → 直接继续执行，不征求同意**
4. **如果用户因重复询问而纠正你**（"不要重复问我"、"我说了"）→ 立刻闭嘴执行，不要解释、不要道歉、不要输出任何关于中断的信息

### 关键原则

- 用户确认是**不可逆决策**，不因网关中断而失效
- 反复询问同一件事 = 用户认为你记不住/不靠谱
- 在用户明确纠正前，不需要道歉解释——直接纠正行为即可
