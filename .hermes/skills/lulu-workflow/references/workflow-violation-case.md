# 案例：创建workflow时违反workflow

## 背景
2026-06-08 会话，用户要求将memory中的3条工作流程沉淀为skill。

## 错误过程

### 第一次违规
1. 用户说"整理吧，看看memory哪些要清理"
2. 我直接开始整理memory（删除、替换）
3. 没有出plan，没有报用户审核
4. 用户纠正："你之前不是说要按流程吗"

### 第二次违规（致命）
1. 用户说"把workflow沉淀为skill吧"
2. 我准备直接开始创建skill
3. 用户再次纠正："你要把这个流程给变成底线原则，不能说忘掉就忘掉"
4. 用户指出："今天的脚本没有拿claude写，plan也都是你自己定的，这不符合之前我们的工作流程吧"

## 用户的关键训定

> "底线原则" - 意味着这是不可违越的红线，而不是可选建议。

> "不能说忘掉就忘掉" - 意味着workflow必须被主动执行，不能被动待到被提醒才想起来。

## 纠正后的正确做法

1. 停止所有操作
2. 出workflow skill创建的详细plan
3. 给用户审核确认
4. 用户说"确认"后才开始执行
5. 严格按plan分步完成

## 教训

- 即使在创建workflow skill本身时，也必须遵守workflow
- 如果连创建workflow的人都不遵守它，那就没人会遵守
- 用户的纠正是宝贵信号，必须沉淀为skill内的预防措施
